## Contraxx

Most agent systems work like this:
you have AGENT.md files in the code base or project your working on.
You have a .claude, .opencode, etc folder with something like
./opencode/agents/frontend-expert.md

you specify the tools it has access to and permissions in the front matter, and the
agent's system prompt in the body of the document.

But what if you want to have a frontend react expert, a frontend vue expert, a frontend jquery expert, etc. Each of these have overlapping needs and concerns. Each also have
unique aspects to them.

You end up having to maintain these overlapping concerns, any time you have a preference
you need changed, if it touches the overlapping concerns, you have to change all
of your agent files.

This is not DRY.

Skills help with this to an extent.
but skill discovery is usually global, or project, not agent level.
And when it is, you end up with this duplication problem again.
agents with overlapping concerns all need to have their skills individually managed.

I solve this with contraxx by breaking breaking agents down into composable contracts.

### Available contracts
- actors
- agents
- roles
- capabilities
- policies
- events
- triggers
- jobs
- workflows


#### Actors
Actors are contextual identities. In Knoxx, each page has an actor associated with it.
Each of these actors is assigned a role that gives them tools and prompts related
to that pages purpose, to facilitate collaboration in that specific context.

Users are also actors. We do this so agent capabilities can map closely to user permissions.


#### Agents

An agent is to a specific task, as an agent is to a context.
An agent is like a persona an actor wears.

People have different personas in different context, that overlay their core identity.
When ever an actor runs, it is running in the context of a specific agent context.

#### Roles

Roles are bundled and composble prompts, capabilities, and policies
that an agent actor can be assigned.
This allows you to bundle sets of related tools, and specific intents behind the use
of those tools into chunks.

Actors and agents can be assigned an arbitrary list of roles.

#### Capabilities

Capabilities are a collection of related agent tools, skills policies, prompts and
related user surfaces, bundled together.

capabilities are things like "slack", "teams","share point"
Sets of tools that are used together to facilitate a specific ability of the agent.
Additionally, each tool is associated with a user surface.

The user surfaces bound to a capability allow for the tools the agent uses
to act on the interface presented to the user,
and for the actor to be aware of the way the user percieves it's actions
in a way it can clearly understand, allowing an agent to be a collaborator, rather than just
a drone you delegate all your work to and walk away.

We all know that if you arenot over seeing the agents work, it's gonna be garbage.

#### Policies

Policies place limitations on the actors. They block access to certain files, they prompt the
user for permission to act, they ensure compliance with mandated work flows.

For example, I have an audit log policy, and a retrospective policy.

The agent *must* justify their actions in a ledger, any signifigantly long turn has a prompt injected to remind it to append an entry to the decision ledger, and if the agent finishes a long turn,
and did not use the ledger tool, a prompt is injected "You must use the append_decision_ledger tol this turn because you used the edit tool X times"

similar deal with the retrospective tool. They are required to score the turn's perceieved difficulty, and if the perceieved difficulty is over a threshold, they must draft a skill file.

If future sessions encounter similar difficulties, related drafted retrospective skills are surfaced via semantic embedding memory.

If they are surfaced X number of times, they are promoted to proper skills.

#### Triggers

A trigger is a set of conditions under which an action can occur.
Usually this is a combination of an event and a conditional expression.
An example is a keyword, an event + a substring present in that event's content.
