 Signal

 You’re right: these are not the emoji/data-labeling tools.

 They are the planned/registered OpenPlanner epistemic-kernel tools: a
 proof/verification pipeline for storing what happened, what is believed, what
 was inferred, what an agent claims it did, and whether that claim later held
 up.

 The core loop is:

 ```text
   obs → fact → inference → attestation → judgment
 ```

 That is much closer to “verify other agents’ claims” than to “label this as
 good/bad.”

 So for your Gemma4 example, the intended workflow would be:

 ```text
   1. Gemma4 says: "I changed X" or "Fact Y is true"
   2. System records that as obs
   3. Auditor agent searches events/graph/files/tools
   4. Auditor emits inference: "Evidence supports/contradicts claim"
   5. Auditor emits attestation: "I performed verification steps A/B/C"
   6. Auditor emits judgment: held / failed / partial
 ```

 That’s a real audit trail, not just a label.

 Countermoves

 I would not make this “Mimo is trusted, therefore judgment is truth.”

 Better framing:

 - Mimo 2.5 Pro is trusted to be a high-quality auditor.
 - The judgment should still be evidence-carrying.
 - The valuable thing is not merely “Mimo said Gemma4 was wrong.”
 - The valuable thing is:

 ```json
   {
     "claim": "Gemma4 said it updated file X",
     "evidence": ["git diff", "file mtime", "tool logs", "session
 transcript"],
     "auditor": "discord_admin_control / mimo-v2.5-pro",
     "verdict": "failed",
     "confidence": 0.91
   }
 ```

 So: use Mimo as the judge, but make Mimo write proof-carrying judgments.

 Also, keep this separate from weak labels:

 - Emoji/reaction labels answer: “Was this output good/useful?”
 - Epistemic judgments answer: “Did this claim/action hold up against
 evidence?”

 They can interact later, but they are different layers.

 Next

 The clean design is:

 1. Let weaker/cheaper agents emit:
     - append-obs for what they saw.
     - append-inference for what they believe.
     - append-attestation for what they claim they did.
 2. Let your sysadmin Mimo 2.5 Pro agent emit:
     - append-judgment after checking files, graph, logs, tool results, or
 external evidence.
 3. Add policy:
     - Only trusted auditor agents can call append-judgment.
     - Normal agents can write obs/inference/attestation, but not final
 judgments.
     - Human approval can override or promote judgments.
 4. Use this as the backbone for agent evaluation:
     - Gemma4 says it completed task.
     - Mimo audits.
     - Judgment gets stored.
     - Over time you get per-model reliability: claim held rate, false
 attestation rate, unsupported inference rate, etc.

 In short: these are the right primitives for a model-claim verification
 pipeline. They are not the labeling system, but they are probably the thing
 you want underneath a serious auditor/reviewer system.

 Frames

 The architecture wants three layers:

 1. Human/weak feedback layer
   This is the existing emoji/reaction label system.
   Purpose:
   ```text
     Was this output useful, funny, wrong, worth attention, or good/bad?
   ```
   Existing examples:
   ```json
     {
       "claim_system": "weak-reaction-v1",
       "source": "knoxx-chat-ui",
       "emoji": "✅",
       "quality": "good"
     }
   ```
 2. Agent annotation / archaeology layer
   This is the missing middle layer.
   Purpose:
   ```text
     What kind of artifact is this?
     What recipe produced it?
     What evidence links this output to a run, tool call, file, prompt, model, or Discord message?
     Is it likely reproducible?
     Should it be promoted?
   ```
   This should not impersonate human labels. It should write a different claim system, for example:
   ```json
     {
       "claim_system": "agent-archeology-v1",
       "label_source": "agent",
       "agent_id": "openplanner_archeologist",
       "model": "mimo-v2.5-pro",
       "confidence": 0.84,
       "labels": [
         "artifact:audio-render",
         "recipe:recoverable",
         "quality:candidate",
         "needs:provenance-check"
       ],
       "evidence_refs": [
         "run:abc",
         "tool_receipt:def",
         "file:outputs/song.wav"
       ]
     }
   ```
 3. Epistemic judgment layer
   This is the proof/verification kernel.
   Purpose:
   ```text
     Did a claim hold up against evidence?
   ```
   Example:
   ```json
     {
       "claim": "Gemma4 changed file X",
       "evidence": ["git diff", "session transcript", "tool receipt"],
       "auditor": "mimo-v2.5-pro",
       "verdict": "failed",
       "confidence": 0.91
     }
   ```
   This belongs in:
   ```text
     obs → fact → inference → attestation → judgment
   ```

 The best design is to let these layers cross-reference each other, not merge them.

 Countermoves

 Do not make append-judgment generally available just because an agent has OpenPlanner access.

 Right now :cap/openplanner grants all seven tools as one bundle. For the system you’re describing, that is too coarse.

 A better split would be:

 ```edn
   :cap/openplanner-read
   [:openplanner.query-graph
    :openplanner.search-events]

   :cap/openplanner-write-observation
   [:openplanner.append-obs
    :openplanner.append-fact]

   :cap/openplanner-write-claims
   [:openplanner.append-inference
    :openplanner.append-attestation]

   :cap/openplanner-judge
   [:openplanner.append-judgment]
 ```

 Then policy can say:

 - normal agents may append obs/inference/attestation;
 - archaeology agents may propose labels and evidence graphs;
 - trusted auditor agents may append judgments;
 - humans/admins may override/promote/reject judgments.

 Also, do not wire the existing ReviewQueuePage and declare victory. It is useful scaffolding, but it is currently a
 generic review queue. The thing you want is closer to an “Audit/Labels” or “Artifact Auditor” surface that can show:

 - artifact/run/message;
 - human labels;
 - agent labels;
 - weak reactions;
 - provenance chain;
 - epistemic claims;
 - attestations;
 - judgments;
 - confidence;
 - evidence refs;
 - promote/reject/merge actions.

 The clean next implementation sequence is:

 1. Split OpenPlanner capabilities
   Add separate Knoxx capability contracts for:
     - read-only OpenPlanner access;
     - obs/fact append;
     - inference/attestation append;
     - judgment append.
   This prevents low-trust agents from writing final verdicts.
 2. Implement the missing OpenPlanner MCP service
   The spec already says it should live at:
   ```text
     orgs/open-hax/tooloxx/services/openplanner-mcp/
   ```
   But that directory currently looks like a stub. Implement:
     - GET /health
     - GET /mcp/schema
     - POST /
     - seven MCP tools
     - Zod validation mirroring promptdb-core
     - OpenPlanner HTTP client
     - per-project/source scoping
 3. Add agent annotation tools separately from epistemic tools
   Do not overload append-judgment for “this artifact seems good.”
   Add something like:
   ```text
     openplanner.append-label-claim
     openplanner.append-artifact-annotation
     openplanner.propose-recipe
     openplanner.link-evidence
   ```
   Or keep it inside existing label APIs but require explicit fields:
     - claim_system
     - label_source
     - agent_id
     - model
     - confidence
     - evidence_refs
     - human_promoted
     - supersedes
 4. Create openplanner_archeologist Knoxx agent
   Its job:
     - search recent runs/artifacts/events;
     - group artifacts by recipe/provenance;
     - propose labels;
     - detect missing provenance;
     - emit inferences/attestations;
     - never emit final judgments unless separately granted.
 5. Create a trusted auditor agent
   This can be the Mimo 2.5 Pro sysadmin-style agent.
   Its job:
     - verify claims against files/logs/events/tool receipts;
     - write evidence-carrying judgments;
     - mark claims as held, failed, or partial.
 6. Mount an Ops “Audit/Labels” UI
   Add a route under Ops, likely:
   ```text
     /ops/audit
   ```
   It should show, per artifact/message/run:
     - human reactions;
     - weak quality labels;
     - agent-proposed labels;
     - epistemic obs/inferences/attestations/judgments;
     - evidence refs;
     - “promote label” / “reject label” / “request audit” actions.
 7. Keep search quality semantics narrow
   Continue using ✅/❌ for search ranking, but do not let agent labels automatically become search quality labels unless

   Good default:
   ```text
     human ✅/❌ affects search immediately
     agent labels are advisory
     auditor judgments affect reliability metrics
     human/admin promotion can convert agent label → trusted quality signal
   ```
