 Signal

 (己, p=0.96) gemma4:e4b audio fits primarily as an Adjudication Agent evidence producer, not as the final judge and not
 as the main extractor.

 In this loop, Gemma’s best role is:

 ```text
   Audio-first bounded adjudicator over short segments:
   A/B audio + evidence.json -> structured observations, hypotheses, confidence, difficulty, next tool actions
 ```

 It should emit normalized evidence observations like:

 ```json
   {
     "schema_version": "fork-tales-evidence-observation/v1",
     "claim_id": "vocal.55.26_61.00.delivery_inflection",
     "domain": "ornament_articulation",
     "target": "lead_vocal",
     "span": { "t0": 55.26, "t1": 61.0 },
     "tool": "gemma4:e4b",
     "role": "adjudicator",
     "observation": {
       "label": "candidate_flatter_less_expressive_than_original",
       "heard_difference": "candidate preserves some lyric identity but loses pitch motion and expressive contour"
     },
     "hypothesis": "render has lyric intelligibility but reduced melodic/delivery fidelity",
     "counterhypotheses": [
       "f0 tracker octave error exaggerates pitch mismatch",
       "candidate is a valid simplified reference rather than a faithful reconstruction"
     ],
     "derived_score": 0.42,
     "confidence": 0.58,
     "difficulty": 0.71,
     "ambiguity_adjustment": 0.85,
     "rationale": "Direct A/B listening plus f0 evidence suggest the words are closer than the melody/delivery.",
     "provenance": [
       "audio:ab-original-candidate.wav",
       "metrics:metrics.json",
       "stt:evidence.json"
     ]
   }
 ```

 Operationally, it sits here:

 ```mermaid
   flowchart LR
       A[Audio segment + candidate] --> B[Extraction tools: STT, f0, spectrogram]
       B --> C[evidence.json]
       A --> D[Gemma4:e4b audio-first audit]
       C --> D
       D --> E[Gemma evidence observations]
       E --> F[Adjudication report / grade]
       F --> G[QC Reviewer]
 ```

 Evidence

 (世, p=0.94) Existing harness path:

 ```bash
   nbb scripts/fork_tales_audio_agent.cljs audit \
     --case-dir <case-dir> \
     --session-id <session-id> \
     --role <role> \
     --original <original.wav> \
     --candidate <candidate.wav> \
     --start <seconds> \
     --duration <seconds> \
     --expected '<lyric clue>'
 ```

 (世, p=0.94) In the current implementation, audit creates:

 - original/candidate slices,
 - local STT outputs,
 - f0/spectrogram metrics,
 - A/B audio,
 - evidence.json,
 - Gemma audio-first response.

 (己, p=0.92) What should change next: Gemma’s raw response should be normalized into the same
 fork-tales-evidence-observation/v1 or judge_scores shape, so it can be weighted by the rubric just like STT, f0,
 spectrogram metrics, and image judges.

 Frames

 (己, p=0.92) Frame 1: Gemma is an audio-aware adjudicator. It is good at saying “these two things differ in this
 perceptual way,” especially when shown the tool evidence.

 (己, p=0.88) Frame 2: Gemma is also a hypothesis generator. It can propose “maybe this is one sustained syllable with
 vibrato tail, not two notes,” which the Planner can turn into a next extraction/render task.

 (己, p=0.83) Frame 3: Gemma is only a weak-to-medium transcription extractor. For lyric identity/timing, Whisper/local
 STT should usually have more weight; Gemma’s lyric observations are useful but lower-authority.

 Countermoves

 (己, p=0.96) Do not use Gemma as the final “does it pass?” judge. The QC Reviewer and rubric gates decide that.

 (己, p=0.94) Do not use Gemma for long full-song A/B judgments. Keep windows short, usually 4–8s, because A/B doubles
 the audio duration and the local runner can fail.

 (己, p=0.93) Do not let Gemma overwrite deterministic f0 evidence. If Gemma says “pitch sounds close” but f0 metrics
 show large contour divergence, the claim is contested and should be flagged.
