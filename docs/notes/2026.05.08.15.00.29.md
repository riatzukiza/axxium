 (己, p=0.9) Yes. Your diagnosis is right: the thing we want is not “two implementations” and probably not a
 src/.../build/... source tree. We want:

 1. one target-neutral implementation;
 2. one declarative EDN intermediary describing how that implementation is exposed;
 3. thin target builders/adapters that consume that EDN and emit Pi/OpenCode entrypoints.

 ```edn
   {:eta-mu.extension/id :opmf-contract-gate
    :eta-mu.extension/ns eta-mu.extensions.opmf-contract-gate

    :eta-mu.extension/capabilities
    [{:id :contract/inject-system-prompt
      :op :opmf-contract-gate/inject-contract-prompt

 ```


 Evidence

 (己, p=0.86) Current shape already partially proves the model:

 - eta_mu.extensions.opmf_contract_gate.cljs is mostly the shared implementation.
 - eta_mu.build.opmf_contract_gate.cljs is the Pi wrapper.
 - eta_mu.build.opencode.opmf_contract_gate.cljs is the OpenCode wrapper.
 - But the OpenCode wrapper has accumulated real behavior: fake message construction, repair enqueue logic, promptAsync
 handling, repair accounting. That is exactly where drift starts.

 (己, p=0.82) The misleading part is naming and responsibility, not compilation itself:

 - ClojureScript does get built/compiled by shadow-cljs.
 - But src/eta_mu/build/... is not build output; it is source code for target adapters.
 - So calling it build makes it look generated or artifact-like when it is actually hand-authored source.

 (己, p=0.78) The bug we just touched is evidence of why this matters: Pi and OpenCode repair delivery differ at the
 target boundary, but the policy around validation, repair budgets, repair prompt construction, and state should not be
 duplicated.

 Frames

 - (己, p=0.87) Architectural frame: we need a target-neutral extension protocol, with Pi/OpenCode as interpreters of the
 same EDN descriptor.
 - (己, p=0.78) Refactor frame: opmf_contract_gate should be split into core operations plus target I/O ports, and the
 target folders should become generated/thin adapter layers.
 - (己, p=0.65) Naming frame: build is causing conceptual damage because it conflates “source that builds adapters” with
 “compiled output.”

 Countermoves

 (己, p=0.82) Don’t make the EDN a giant programming language. It should describe bindings, capabilities, target event
 names, delivery modes, and permissions. Real control flow should stay in CLJS.

 (己, p=0.78) Don’t fully erase target differences. Pi and OpenCode really do have different lifecycle semantics. The
 shared layer should own policy; target adapters should own transport.

 (己, p=0.76) Don’t hand-maintain two wrappers once the descriptor exists. Generate them or make them tiny generic
 interpreters, then add a test that fails if target code imports validation/repair internals directly instead of going
 through the shared op table.

 A cleaner layout could be:

 ```text
   packages/eta-mu-extensions/
     resources/eta_mu/extensions/
       opmf-contract-gate.edn

     src/eta_mu/extensions/opmf_contract_gate/
       core.cljs          ; pure-ish shared contract logic
       ops.cljs           ; named operations exposed to descriptor
       state.cljs         ; shared state/config/artifact code

     src/eta_mu/targets/
       pi.cljs            ; generic Pi adapter/emitter
       opencode.cljs      ; generic OpenCode adapter/emitter

     target/generated/
       pi/opmf_contract_gate.cljs
       opencode/opmf_contract_gate.cljs

     dist/
       pi/...
       opencode/...
 ```

 (己, p=0.74) The key contract is: target files may normalize events and call ops; they may not implement output-contract
 semantics.
